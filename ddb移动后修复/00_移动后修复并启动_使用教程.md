# DolphinDB 3.00.6 + Starfish 移动后路径修复使用教程

## 1. 用途

这套脚本用于解决以下问题：

1. DolphinDB 已经启动和使用过，根目录下已经生成 `local8848`。
2. 随后把整个程序目录改名、复制到其他目录，或者移动到其他盘符。
3. `local8848` 的 DFS 元数据仍然保存旧的绝对路径，导致 DolphinDB 数据库无法打开，Starfish 无法读取 `facplf_*` 数据库。

脚本会自动识别 BAT 当前所在的新根目录，把 DFS 元数据中的旧根路径更新为新根路径，然后调用原来的启动器。

## 2. 文件位置

下面三个文件必须放在 DolphinDB 程序根目录：

```text
移动后修复并启动_DDB.bat
_repair_local8848_path.ps1
启动_DDB.bat
```

正常使用时只需要双击：

```text
移动后修复并启动_DDB.bat
```

不要单独移动 BAT，也不要删除同目录的 `_repair_local8848_path.ps1`。

## 3. 适用范围

当前脚本按以下环境设计和验证：

```text
DolphinDB Server V3.00.6
Windows 单机模式 mode=single
localSite=localhost:8848:local8848
数据目录名称为 local8848
当前这套 DolphinDB + Starfish 目录结构
```

支持：

- 修改英文目录名称。
- 从一个英文目录移动到另一个英文目录。
- 从一个盘符移动到另一个盘符。
- 新旧路径长度不同。
- 已有多个 DFS 数据库和大量业务数据。
- 没有 `local8848` 的全新环境，此时跳过迁移并直接启动。

完整路径必须使用 ASCII 字符，建议只使用：

```text
英文字母、数字、空格、下划线、短横线
```

示例：

```text
C:\DDB306
D:\Tools\DDB_Server
E:\Project\DDB_Test_01
```

不要使用中文程序路径。实测 V3.00.6 或其插件可能在中文路径下停在 `startup::loadConfig` 后退出，脚本会提前拒绝这种路径。

## 4. 标准操作步骤

### 第一步：安全关闭 DolphinDB

移动目录之前，先双击：

```text
安全关闭_DDB.bat
```

确认 DolphinDB 已停止。不要在 `dolphindb.exe` 正在运行时复制、改名或迁移元数据。

### 第二步：整体移动或改名

必须移动整个程序目录，包括：

```text
local8848
modules
plugins
starfish
启动_DDB.bat
安全关闭_DDB.bat
移动后修复并启动_DDB.bat
_repair_local8848_path.ps1
```

不要只复制 `local8848` 中的一部分。

### 第三步：双击迁移启动器

在新目录双击：

```text
移动后修复并启动_DDB.bat
```

Windows 弹出管理员权限确认时选择“是”。

脚本依次执行：

```text
取得 BAT 当前所在的新根目录
检查 dolphindb.exe 和端口 8848
扫描 local8848\storage\CHUNK_METADATA
识别元数据中保存的旧根目录
备份整个 CHUNK_METADATA
逐字节替换旧根目录
校验旧路径已经清零
调用原来的 启动_DDB.bat
启动 DolphinDB 和 Starfish
```

### 第四步：检查结果

至少检查：

1. DolphinDB Web 可以打开。
2. 数据库列表正常显示。
3. 随机打开几个原有数据库和表。
4. Starfish 首页可以进入。
5. Starfish 的数据管理、因子、模板或策略页面可以读取原有内容。
6. 使用一段时间后，再执行一次安全关闭和重新启动。

## 5. 脚本不是固定偏移修改

脚本没有写死旧目录名称、字节偏移或替换次数。

它递归扫描：

```text
local8848\storage\CHUNK_METADATA
```

通过以下物理路径特征识别旧根目录：

```text
\local8848\storage\CHUNKS\
\local8848\storage\DATABASE\
```

识别旧根目录后，再对元数据文件执行精确的 UTF-8 字节序列替换。因此新旧路径长度可以不同，路径出现次数也不固定。

目前实测环境中，真正包含旧路径并被修改的是：

```text
local8848\storage\CHUNK_METADATA\editlog.0
```

如果以后 `CHUNK_METADATA` 中出现其他 editlog 或 checkpoint，脚本也会扫描它们，但只修改确实包含旧路径字节的文件。

## 6. 备份在哪里

只有真正需要替换路径时，脚本才会创建备份。

备份根目录：

```text
当前程序根目录\_path_migration_backup
```

里面通常包含：

```text
20260719_201405_CHUNK_METADATA
20260719_201405_migration_report.txt
```

其中：

- `时间_CHUNK_METADATA`：修改前的完整元数据备份，可以用于恢复。
- `时间_migration_report.txt`：迁移报告，不是数据库备份。

实际运行使用的元数据仍然是：

```text
local8848\storage\CHUNK_METADATA
```

不要把 `_path_migration_backup` 误认为当前运行目录。

## 7. 迁移报告怎么看

成功报告示例：

```text
OldRoot=C:\Users\likai\Desktop\ddb306_data_move_old
NewRoot=C:\Users\likai\Desktop\ddb306_data_move_new_long
ChangedFile=...\local8848\storage\CHUNK_METADATA\editlog.0
Replacements=515
Result=SUCCESS
ChangedFiles=1
TotalReplacements=515
```

关键字段：

- `OldRoot`：元数据原来保存的程序根目录。
- `NewRoot`：BAT 当前所在的新根目录。
- `ChangedFile`：实际被修改的元数据文件。
- `Replacements`：该文件替换了多少处旧路径。
- `Result=SUCCESS`：二进制替换和最终检查成功。

## 8. 出错时怎么办

### 提示 DolphinDB 正在运行

先执行：

```text
安全关闭_DDB.bat
```

确认进程和端口停止后重新运行迁移 BAT。

### 提示端口 8848 被占用

说明已有 DolphinDB 或其他程序正在监听 `8848`。不要强行迁移，先确认并关闭占用端口的程序。

### 提示检测到多个旧存储根目录

脚本会主动停止，防止误改多磁盘 `volumes` 或多根目录环境。不要直接删除报错，需要先确认该环境是否配置了多个数据盘。

### 路径替换过程中失败

脚本会尝试使用本次创建的 `时间_CHUNK_METADATA` 自动恢复已修改的元数据，并在迁移报告中记录：

```text
Result=ROLLED_BACK
```

如果需要人工恢复：

1. 确认 DolphinDB 已完全关闭。
2. 保留当前失败现场，不要继续重复启动。
3. 找到本次时间对应的 `_path_migration_backup\时间_CHUNK_METADATA`。
4. 用该备份恢复 `local8848\storage\CHUNK_METADATA`。

人工恢复前建议再复制一份整个当前环境，避免覆盖唯一现场。

## 9. 已完成的实际验证

测试不是只在空数据库上进行。

独立测试副本中完成了：

```text
原有数据库：11 个
新增 HASH 分区数据库：5 个
最终数据库总数：16 个
新增数据：1,000,000 行
local8848 活动文件：715 个
旧路径命中：515 次
实际修改文件：editlog.0
```

改名迁移后验证：

- 5 个新增数据库每个仍为 200,000 行。
- 总计 1,000,000 行完整存在。
- `id`、`qty`、`price` 校验和与迁移前一致。
- 数据库总数仍为 16。
- Starfish 四个模块能够重新加载。
- Starfish 核心 `facplf_*` 数据库正常存在。
- 再次关闭后，旧路径为 0 次，新路径增加到 540 次。
- 第二次启动后，100 万行和 Starfish 仍然正常。

这说明后续新增元数据会自动使用新根目录，不需要每次启动都重复替换。

## 10. 限制和注意事项

当前没有宣称以下环境已经验证：

- 集群、控制节点、数据节点模式。
- 多磁盘 `volumes`。
- 其他 DolphinDB 大版本。
- 中文程序路径。
- 已生成特殊压缩或带额外校验格式 checkpoint 的长期生产环境。

对长期运行、数据非常重要的环境，推荐先完整复制整个环境，在副本中改名并执行迁移，然后核对数据库、表、分区、行数和业务校验和。

不要删除备份，直到新位置已经连续完成至少一次：

```text
启动 -> 读取数据库 -> 使用 Starfish -> 安全关闭 -> 再次启动
```
